1、JVM的位置（动态度量JVM）

JDK是java标准的开发包，它提供了编译、运行java程序所需要的各种工具和资源，包括Java编译器、Java运行时环境，以及常用的Java类库。

JDK目录：

JRE、Java运行时环境、用于解释执行Java字节码文件。普通用户只需安装JRE来运行Java程序。

JVM，Java虚拟机，是JRE的一部分。他是JAVA实现跨平台最核心的部分。负责解释执行字节码文件，是可运行字节码文件的虚拟机。

当JAVA编译器编译Java程序时，生成的是与平台无关的字节码，这些字节码只面向JVM。不同平台的JVM都是不同的，但是他们提供了相同的接口。

4、双亲委派机制

安全机制（优先让父类加载的机制）

  1. 类加载器收到类加载请求（为了安全，不知道父加载器中有没有该类 .class文件，需要父类确认是否能够加载该类）
  2. 将这个请求向上委托给父类加载器去完成，一直向上委托，直到启动类加载器
  3. 启动类加载器检查是否能够加载当前这个类（是否存在该.class文件），能加载就结束，使用当前加载器加载。否则通知子类加载器加载

总结来说：

当一个类收到了类加载请求，他首先不会尝试自己去加载这个类，而是把这个请求委托给父类去完成。每一个层次类加载器都是如此，因此所有的加载请求都应该传送到启动加载器中，只有当父类加载器反馈自己无法完成这个请求的时候（在他的加载路径下没有找到所需的加载的Class），子类加载器才会尝试自己去加载。

从下图中我们就更容易理解了，当一个Hello.class这样的文件要被加载时。不考虑我们自定义类加载器，首先会在AppClassLoader中检查是否加载过，如果有那就无需再加载了。如果没有，那么会拿到父加载器，然后调用父加载器的loadClass方法。父类中同理也会先检查自己是否已经加载过，如果没有再往上。注意这个类似递归的过程，直到到达Bootstrap classLoader之前，都是在检查是否加载过，并不会选择自己去加载。直到BootstrapClassLoader，已经没有父加载器了，这时候开始考虑自己是否能加载了，如果自己无法加载，会下沉到子加载器去加载，一直到最底层，如果没有任何加载器能加载，就会抛出ClassNotFoundException。那么有人就有下面这种疑问了？

为什么要设计这种机制
这种设计有个好处是，如果有人想替换系统级别的类：String.java。篡改它的实现，在这种机制下这些系统的类已经被Bootstrap classLoader加载过了（为什么？因为当一个类需要加载的时候，最先去尝试加载的就是BootstrapClassLoader），所以其他类加载器并没有机会再去加载，从一定程度上防止了危险代码的植入。

优势：

避免类重复加载：当父类中ClassLoader已经加载了该类的时候，就不需要子ClassLoader再加载一次了

11、堆

Heap，一个JVM只有一个堆内存，堆内存的大小是可以调整的。

类加载器读取了类文件后，一般会把什么放到堆中？类、方法、常量、变量，保存所有引用类型的真是对象。

堆内存中还要细分为三个区域：

  • 新生区 New （伊甸园区Eden Space、幸存区0区、幸存1区）
  • 养老区 Old （）
  • 永久存储区 Perm

GC垃圾回收主要是在 伊甸园区和养老区

假设堆内存满了，OOM（OutOfMemoryError：Java heap space），堆内存不够。

JDK8之后，永久存储区改为元空间。

新生区

  • 类：诞生和成长的地方，甚至死亡
  • 伊甸园区：所有的对象都是在伊甸园区new出来的
  • 幸存区（0,1）

老年区

经过研究，99%的对象都是临时对象。

永久区

这个区域常驻内存的。用来存放JDK自身携带的Class对象。Interface元数据，存储的是Java运行时的一些环境或类信息，这个区域不存在垃圾回收，关闭JVM虚拟机就会释放这个区域的内存。

一个启动类，加载了大量的第三方jar包。Tomcat部署了太多的应用，大量动态生成的反射类。不断被加载。知道内存满，就会出现OOM

  • jdk1.6之前：永久代，常量池在方法区；
  • jdk1.7：永久代，但是慢慢的退化了，去永久代，常量池在堆中
  • jdk1.8：无永久代，常量池在元空间中。

方法区也叫作非堆，方法区是共享的，存储一些静态变量、常量、类信息、运行时常量；因此在物理上在堆空间中，逻辑上称为方法区；方法区中又包含了常量池

元空间逻辑上存在于Heap中，但是物理上不存在于堆中。可以通过程序打印信息计算。

13、GC垃圾回收

JVM在进行GC时，并不是对这三个区域统一回收。大部分时候，回收新生区

  • 新生区
  • 幸存区（from，to）
  • 老年区

GC两种类：轻GC（新生区，幸存区），重GC（全局）

1、标记清除法

  • 缺点：两次扫描，严重浪费时间，会产生内存碎片
  • 优点：不需要额外的空间

2、标记压缩法

为了防止内存碎片的产生，对标记清除之后的空间再进行一次扫描，将对象移动到连续区域。

可以继续优化，没经历多次GC标记清除之后，进行一次压缩，减少压缩的频率，也能有效防止碎片的产生。

3、复制算法

1、每次GC都会将Eden活的对象移动到幸存区中，一但Eden区被GC后，就会变空（没有用的对象被回收，可以存活的对象被放入幸存区中的To区）

2、每次GC，幸存区中的对象的会将from中的可以存活的对象赋值到to区，此时，from区为空。则from区转换为to区，to区转换为from区。

​ 谁空谁是to区

3、当一个对象经历了15次GC之后，还是没有死，则会进入养老区

  • 幸存区有两个就是为了解决对象放不下内存碎片化的问题。因为永远有一个空的幸存区可以放。

  • 优点：没有内存碎片

  • 缺点：多一块幸存区空间，to空间永远是空的，造成空间浪费。此外，假设from区中的对象全部存活，还需要将对象从from区复制到to区，性能消耗大

  • 复制算法最佳使用场景：对象存活度较低的时候；新生区使用合适~

5、总结

内存效率：复制算法 > 标记清除算法 > 标记压缩算法 （时间复杂度）

内存整齐度：复制算法 = 标记压缩算法 > 标记清除算法

内存利用率：标记压缩算法 = 标记清除算法 > 复制算法

1、什么是JMM？

Java Memory Module

2、他是干什么的？：官方、博客

作用：缓存一致性协议，用于定义数据读写的规则

JMM定义了线程工作内存和主内存之间的抽象关系：线程之间的共享变量存储在主内存（Main Memory），每个线程都有一个私有的本地内存（Local Memory）

CPU中运行的线程从主村中拷贝共享对象object到他的CPU缓存中，把对象object的count变量改为2.但这个并更对运行在右边的CPU中的线程不可见，因为这个更改还没有flush到主存中；要解决共享对象可见性问题，我们可以使用Java volatile关键字或者加锁

Java内存区域详解

1、对象的创建

  • 类加载检查

    虚拟机遇到new指令，首先检查new指令的参数是否能在常量池中定位到这个类的符号引用，并检查这个符号引用代表的类是否被加载过。如果没有，先执行相应的类加载过程。

  • 分配内存

    类检查通过后，虚拟机将会为新生对象分配内存。分配方式有 “指针碰撞” 和 “空闲链表” 两种。分配方式由内存是否规整决定，内存是否规整由垃圾回收算法决定。

  • 初始化零值

    对内存空间初始化零值

  • 设置对象头

    为对象添加头信息。例如：对象时哪个类的实例、如何找到类元数据的信息、对象的hashCode、GC分代年龄。

  • 执行init方法

    在虚拟机角度看，创建对象已经完成，但是对于用户来讲，对象的字段还没有进行初始化。

2、对象内存布局

  • 对象头
  • 实例数据
  • 对齐内存

3、对象的访问定位

  • 句柄：句柄方式优势在于reference中存储的值不需要改变，是一个稳定的地址，对象在堆中移动时，只需要改变句柄中 的实例数据指针。

  • 直接指针：这种方式在移动对象时，需要改变reference中值。但是访问的速度快。

JVM垃圾回收详解

1、常见面试题

  • 如何判断对象是否死亡？

    • 引用计数法：对于引用计数法中，当一个对象的引用计数为0时，表示这个对象不再被使用，也就是说该对象已经死亡，在GC触发时，该对象所占用的内存将被回收。

    • 可达性分析算法：该算法中，如果一个对象存在从GC Roots到该对象的引用链，则该对象是从GC Roots可达的，如果不可达，则在GC时将被回收。

  • 简单介绍一下强引用、软引用、弱引用、虚引用

    • 目的：可以让程序猿通过代码的方式决定某些对象的生命周期；第二是有利于JVM进行垃圾回收，引用定义了对象的生命周期。

    • 强引用：通常使用的大部分引用都是强引用，当一个对象存在强引用，垃圾回收器是不会对该对象进行回收的

    • 软引用：软引用描述有用但是不是必须的对象，在java中使用java.lang.ref.Softreference类表示。对于软引用关联的对象，只有在内存不足时JVM才会回收此对象。

    • 弱引用：当JVM进行垃圾回收时，无论内存是否充足，都进行回收。

    • 虚引用：不决定对象的生命周期，主要用来判断那个对象将要被回收。

      虚引用必须和引用队列（ReferenceQueue）联合使用。当垃 圾回收器准备回收一个对象时，如果发现它还有虚引用，就会在回收对象的内存之前，把这个虚引用加入到与之关联的引用队列中。程序可以通过判断引用队列中是 否已经加入了虚引用，来了解被引用的对象是否将要被垃圾回收。

  • 虚引用与软引用和弱引用的区别？使用软引用能带来的好处？

    • 区别：虚引用必须和引用队列结合使用。

    • 使用软引用的好处：可以加速JVM对垃圾回收的速度，可以维护系统的运行安全，防止内存溢出。

      举例：比如说有一个方法，方法中创建一个具有强引用的对象，而这个对象只是在代码的前一行使用了，而Java中没有手动释放资源，也没有手动设置为null 的习惯，但是显然这个对象没有用了，并且它只能在方法执行完毕才能释放空间。但是设置了软引用之后，如果方法结束前执行了垃圾回收，此时，节省了空间，缩短了该对象的生命周期。

  • 如何判断一个常量是废弃常量

    • 假如在字符串常量池中存在字符串 “abc”，如果当前没有任何 String 对象引用该字符串常量的话，就说明常量 “abc” 就是废弃常量，如果这时发生内存回收的话而且有必要的话，”abc” 就会被系统清理出常量池了、

  • 如何判断一个类是无用类

    • 该类所有的实例都已经被回收，也就是 Java 堆中不存在该类的任何实例。

    • 加载该类的 ClassLoader 已经被回收。

    • 该类对应的 java.lang.Class 对象没有在任何地方被引用，无法在任何地方通过反射访问该类的方法。

      虚拟机可以对满足上述的无用类进行回收，但不是必须的

  • 垃圾回收算法有哪些，各自的特点？

    • 标记-清除算法

      存在两个阶段：标记，清楚

      特点：

      ​ 1、效率低

      ​ 2、会产生内存碎片

    • 标记-复制算法

      分为from区和to区，新生代使用

      特点：

      ​ 1、内存利用率低，不会超过50%

      ​ 2、效率高

      ​ 3、不会产生内存碎片

    • 标记-整理算法

      标记、清楚、移动并回收。老年代使用

      特点：

      ​ 1、效率低

      ​ 2、不存在内存碎片

      ​ 3、内存利用率高

    • 分代收集算法

  • HtSpot为什么要分为新生代、和老年代。

    分为新生代和老年代，可以更灵活的根据对象的存活周期区选择合适的垃圾回收算法。便于管理内存。

    比如，在新生代采用“标记-清除，标记-整理”算法，在老年代采用“标记-复制”算法

  • 常见的垃圾回收器有哪些

    • Serial收集器

    • ParNew收集器

    • Parallel Scavenge收集器

      这是JDK1.8默认收集器，他的关注点是，吞吐量（高效的利用CPU）。CMS等垃圾收集器关注点更多的是用户线程的停顿时间（提高用户体验）

      用户吞吐量：就是CPU中用于运行用户代码的时间与总消耗的 时间的比值。

    • Serial Old收集器

    • Parallel Old收集器

    • CMS收集器

      CMS收集器是一种以获取最短回收停顿时间为目标的收集器。非常符合在注重用户体验的应用上使用。

      CMS收集器是HotSpot虚拟机第一款真正意义上的并发收集器，他第一次实现了让垃圾收集线程和用户线程同时工作。

      **优点:**并发收集、低停顿

      缺点：对cpu资源敏感、无法处理浮动垃圾、使用的回收标记算法导致收集结束时会有大量的空间碎片产生。

      步骤：

      • 初始标记：暂停其他所有线程，标记直接与root相连的对象。
      • 并发标记：GC与用户线程并发执行。标记可达对象。并跟踪记录期间发生引用更新的地方。
      • 重新标记：对并发标记中变动部分进行标记，时间远远短于并发标记阶段
      • 并发清除：开启用户线程，同时GC线程开始对未标记的区域做清扫。

      G1收集器

      是面向服务器的垃圾收集器，主要针对多处理器，大内存，的机器。低停顿、高吞吐量。

      JDK1.7的重要进化特征。

      特点：

      • 并发与并行：GC线程与Java线程并发执行
      • 分带收集：保留分代概念
      • 空间整合：整体上基于“标记-整理”，局部上基于“标记-复制”算法
      • 可预测停顿：使用者可以指定时间片段

  • Minor GC 和 Full GC有什么不同？

    1、触发的条件不同

    ​ 当新创建的对象加入Eden区时，如果Eden没有空间可存入新的对象，则会触发一次轻GC，并将Eden中可 以存活的对象晋升。

    ​ 当从新生区晋升的对象比老年区剩余空间大时，会触发Full GC，或者永久代满时，或者调用System.gc()

    2、垃圾回收的区域不同

    ​ Minor GC作用在新生代，Full作用在老年代。

2、常见调优参数

  • 动态年龄增长
    • HotSpot遍历所有的对象时，按照年龄从小到大排序，然后对器所占内存进行累加，当累计的内存大小超出了survivor区的一般时，去这个年龄和MaxTenuringThreshold中较小的作为晋升老年区年龄的阈值。
    • 这样做的目的就是保证survivor区有一般是空的，便于复制算法的交换。超过这个年龄不在需要在幸存区中增加年龄，直接进入老年区。

3、GC策略总结

按照HotSpot VM的serial GC 的实现总计：

1、Minor GC（轻GC、YoungGC）

  • 新创建的对象首先进入 新生代 的 Eden区，如果该对象的太大放不下 Eden区，则将触发MinorGC，将Eden区中的可以存活的对象转入 Survivor 区，如果发现Survivor区放不下，则会通过分配担保机制把新生代的对象提前转入老年代。

  • 新创建的对象首先分配 新生代 的 Eden区，如果Eden区已经满了，此时放不下了。则会触发一次轻GC—-MinorGC，将可以存活的对象转入Survivor区。

2、Full GC（重GC）

  • 新创建的对象首先分配 新生代 的 Eden区，如果发现晋升到 老年代 的对象大小比 老年代剩余空间大，则会直接触发 重GC===FullGC

  • 如果有 Perm gen（永久代）的话，要在Perm gen分配空间，但是已经没有足够的空间了，也会触发一次Full GC

  • 或者 System.gc()调用时，触发Full GC

3、Old GC、Major GC

4、整堆收集：

  • 收集整个Java堆和方法区

5、空间分配担保：

  • 是为了确保在 Minor GC 之前老年代本身还有容纳新生代所有对象的剩余空间。

4、判断对象死亡

在垃圾回收之前，通常需要判断对象是否死亡？当对象不存在引用时，表示该对象死亡。

4.1、引用计数法

给对象中添加一个引用计数器，每当有一个地方引用他，计数器加1；当引用失效，计数器减1；在何时计数器为0时的对象就是不可能在被使用。

这个方法非常简单，效率高，但是目前主流的虚拟机中并没有选择这个算法来管理内存，其主要原因是他很难解决对象之间相互循环引用的问题。

4.2、可达性分析算法

这个算法的思想是通过一系列的称为 “GC Roots” 的对象作为起点，从这些节点开始搜索，节点所走过的路径称为引用链，当一个对象到GC Roor没有任何引用链相连的话，则证明此对象是不可用的，需要被回收。

下图中，6~10之间虽然有引用关系，但是他们到GC Roots不可达，因此需要被回收的对象。

哪些对象可以作为GC Roots呢？

  • 虚拟机栈中引用的对象（栈中的本地变量表）
  • 本地方法栈中引用的对象（Native 方法）
  • 方法区中静态属性引用的对象
  • 方法区中常量引用的对象
  • 所有被同步锁持有的对象

5、常量池理解

常量池、运行时常量池、字符串常量池

  • JDK1.7 之前运行时常量池逻辑包含字符串常量池存放在方法区, 此时 hotspot 虚拟机对方法区的实现为永久代
  • JDK1.7 字符串常量池被从方法区拿到了堆中, 这里没有提到运行时常量池,也就是说字符串常量池被单独拿到堆,运行时常量池剩下的东西还在方法区, 也就是 hotspot 中的永久代 。
  • JDK1.8 hotspot 移除了永久代用元空间(Metaspace)取而代之, 这时候字符串常量池还在堆, 运行时常量池还在方法区, 只不过方法区的实现从永久代变成了元空间(Metaspace)

6、CMS收集器

CMS收集器是一种以获取最短回收停顿时间为目标的收集器。非常符合在注重用户体验的应用上使用。

CMS收集器是HotSpot虚拟机第一款真正意义上的并发收集器，他第一次实现了让垃圾收集线程和用户线程同时工作。

**优点:**并发收集、低停顿

缺点：对cpu资源敏感、无法处理浮动垃圾、使用的回收标记算法导致收集结束时会有大量的空间碎片产生。

步骤：

  • 初始标记：暂停其他所有线程，标记直接与root相连的对象。
  • 并发标记：GC与用户线程并发执行。标记可达对象。并跟踪记录期间发生引用更新的地方。
  • 重新标记：对并发标记中变动部分进行标记，时间远远短于并发标记阶段
  • 并发清除：开启用户线程，同时GC线程开始对未标记的区域做清扫。

7、G1收集器

是面向服务器的垃圾收集器，主要针对多处理器，大内存，的机器。低停顿、高吞吐量。

JDK1.7的重要进化特征。

特点：

  • 并发与并行：GC线程与Java线程并发执行
  • 分带收集：保留分代概念
  • 空间整合：整体上基于“标记-整理”，局部上基于“标记-复制”算法
  • 可预测停顿：使用者可以指定时间片段

2、类加载过程

  • 加载

    1. 通过全类名获取定义此类的二进制字节流
    2. 将字节流所代表的静态存储结构转换为方法区的运行时数据结构
    3. 在内存中生成一个代表该类的 Class 对象，作为方法区这些数据的访问入口

  • 验证

    1. 文件格式验证：是否符合ClassFile文件格式、版本号、常量池中的常量是否支持
    2. 元数据验证：对字节码描述信息进行语义分析（父类、继承父类是否可行）
    3. 字节码验证：
    4. 符号引用验证

  • 准备

    准备阶段是正式为类变量分配内存并设置类变量初始值的阶段，这些内存都将在方法区中分配。

    1. 分配类变量内存，不包括实例变量。实例变量在对象创建时在Java堆中分配。
    2. 1.7之前静态变量、字符串常量在永久代中，1.7之后再Java堆中。
    3. 为静态变量赋予0值

  • 解析

    解析阶段是虚拟机将常量池内的符号引用替换为直接引用的过程。

    综上，解析阶段是虚拟机将常量池内的符号引用替换为直接引用的过程，也就是得到类或者字段、方法在内存中的指针或者偏移量。

  • 初始化

    阶段是执行初始化方法 <clinit> ()方法的过程，是类加载的最后一步，这一步 JVM 才开始真正执行类中定义的 Java 程序代码(字节码)。

    对于<clinit> () 方法的调用，虚拟机会自己确保其在多线程环境中的安全性。因为 <clinit> () 方法是带锁线程安全，所以在多线程环境下进行类初始化的话可能会引起多个进程阻塞，并且这种阻塞很难被发现。

    类初始化的5个场景：

    • 当遇到 new 、 getstatic、putstatic 或 invokestatic 这 4 条直接码指令时
    • 使用 java.lang.reflect 包的方法对类进行反射调用时。如 Class.forname("..."), newInstance() 等等。如果类没初始化，需要触发其初始化。
    • 当虚拟机启动时，用户需要定义一个要执行的主类 (包含 main 方法的那个类)，虚拟机会先初始化这个类。
    • MethodHandle 和 VarHandle 可以看作是轻量级的反射调用机制，而要想使用这 2 个调用， 就必须先使用 findStaticVarHandle 来初始化要调用的类。
    • 当一个接口中定义了 JDK8 新加入的默认方法（被 default 关键字修饰的接口方法）时，如果有这个接口的实现类发生了初始化，那该接口要在其之前被初始化。

  • 卸载

    该类的Class对象被GC。

    条件：

    1. 该类的所有对象都被GC
    2. 给类没有被引用
    3. 该类的类加载器的实例被GC

    所以，在 JVM 生命周期内，由 jvm 自带的类加载器加载的类是不会被卸载的。但是由我们自定义的类加载器加载的类是可能被卸载的
